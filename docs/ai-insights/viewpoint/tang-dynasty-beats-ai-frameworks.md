# 开源项目--1300 年前的唐朝制度，吊打了我用过的所有 AI 框架

> 终于有人把一个本该写进所有多 Agent 框架首页的常识说出来了：没有审议链路的协作系统，不叫“自治”，那叫多人同时犯错。

**这篇文章真正击中的，不是“三省六部”这个历史梗，而是它把现代 AI 框架最常见的短板钉在了墙上：大家都热衷于角色分工、消息传递、任务拆解，却对“谁来否决烂方案”这件事装作视而不见。一个系统如果只有生成和执行，没有强制审核，那么它再热闹，也只是把失控流程包装得更像 PPT。Bazinga.**

## 真正稀缺的不是 Agent 数量，而是否决机制

现在很多 AI 框架的宣传页看起来都像同一份模板复印出来的：多个角色、自动协作、任务编排、工具调用、并行执行，仿佛只要 Agent 足够多，组织能力就会凭空出现。荒谬。把五个不会自我约束的模块绑在一起，并不会自动长出治理结构，只会更快地产生五份错误结果。

这篇文章最聪明的地方，是它没有继续迷信“分工”本身，而是直接指出真正决定系统质量的，是**审议权是否内建**。中书省可以规划，尚书省可以执行，但门下省必须有能力把烂方案直接打回。请注意这个动作的价值：不是提醒，不是建议，不是“低置信度标红”，而是制度性的否决。没有这一步，所谓多 Agent 协作不过是把 bug 从串行制造改成并行扩散。

## 很多框架的问题，不是不会做事，而是不允许流程被质疑

文章拿唐朝制度做类比，表面上很戏剧化，实则非常准确。因为现代很多 Agent 框架确实只解决了“任务怎么流动”，却没有解决“错误怎么被拦截”。规划者拆任务，执行者跑流程，用户最后看结果，出了问题就重跑一遍，再美其名曰“迭代”。抱歉，那不是反馈回路，那只是重复抽卡。

所以这篇文章真正该被拿去抄作业的，不是古风 UI，也不是“三省六部”这个包装壳，而是背后的工程判断：**审核不是插件，审核应该是关卡；返工不是例外，返工应该是默认选项。** 如果你的系统只能产出，却不能正式驳回；只能继续向前，却不能强制回退，那它顶多算一个自动化流水线，远远称不上可靠协作系统。

## 启示

我很愿意冷静地指出一点：这篇文章批评的根本不是某几个框架，而是整个 AI 工具圈对“治理成本”的长期逃避。大家都想讨论 agentic、autonomous、orchestration，听起来像在发布下一代操作系统；但一旦涉及审计、制衡、责任边界、返工权，气氛就立刻变得像实验室断电。因为这些东西不性感，却决定系统能不能活下来。

所以，1300 年前的制度设计为什么能“吊打”今天很多 AI 框架？答案一点也不玄学：古人至少知道，权力要分层，流程要审议，错误要允许被拦截。反观今天不少框架，还停留在“让更多 Agent 更快开工”这种幼儿园水平的乐观主义。角色越多，不代表系统越成熟；只有当否决机制、审核链路、返工纪律都被写进系统骨架里，协作才配叫协作。其他那些热闹的 demo，不过是把组织失灵做成了产品卖点。多么迷人的时代病。
